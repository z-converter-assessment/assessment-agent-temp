# ADR-0001: OS별 2트리를 대칭 중복으로 유지

상태: 채택

## 맥락

수집기는 커널 인터페이스를 직접 다룬다 — Linux는 procfs/sysfs 텍스트 파싱, Windows는 Win32 API/IOCTL/Native API/perflib. 두 OS의 원천 성격이 근본적으로 달라, 하나의 크로스플랫폼 추상화로 감싸면 추상화 경계가 얇은 곳(공통)과 새는 곳(OS 전용 신호)이 뒤섞여 정확성과 로드 안전성이 흐려진다.

## 결정

Linux(`src/`, `include/`)와 Windows(`windows-agent/src/`, `windows-agent/include/`)를 별개 소스 트리로 두고, 파일 레이아웃·네이밍·필드셋을 대칭으로 유지한다. 공통 로직도 트리 간 공용화하지 않는다. 스키마·로직 변경은 양쪽에 반영한다.

## 결과

- 유지비가 2배다. 한쪽 신호 추가는 다른 쪽도 실측 발행하거나 측정불가 null로 필드셋을 맞춰야 한다.
- 대가로 각 OS의 커널 인터페이스를 직접·정확하게 다루고, Windows의 로드 안전성(하드임포트 회피, [0005](0005-nt52-load-guard.md))을 트리 단위로 격리한다.
- 트리 간 드리프트는 CI 계약 검증(스키마+어휘 4종 emit)과 필드셋 대칭 규칙으로 막는다.
- 트리 내 중복 제거(같은 트리의 rmrf 사본 통합 등)는 정당하나, 트리 간 공용화는 이 결정의 대상이 아니다.

## 대안

단일 크로스플랫폼 추상화(HAL) — OS 전용 신호(PSI, perflib saturation, NT5.2 폴백)에서 추상화가 새고, 로드 가드를 공통 코드에 섞으면 위반 탐지가 어려워져 기각.
